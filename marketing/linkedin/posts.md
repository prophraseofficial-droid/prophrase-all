## Post 001 — Scope changed

**Creative:** `creatives/scope-changed-friday-didnt.png`

Scope changed. Friday didn't.

This is where work messages get awkward.

You need to protect the deadline without sounding unhelpful. So the first draft
usually becomes too apologetic, too defensive, or far too long.

Try this structure instead:

1. Acknowledge the new request.
2. State the constraint plainly.
3. Offer a decision: move the date or reduce the scope.

Rough:

“can’t add all this by friday, it wasn’t in scope”

Ready:

“I can help with the additional work, but it sits outside the scope we agreed
and won’t fit the Friday timeline. We can either move the date or decide which
current items to replace.”

ProPhrase turns the rough version into something clear, firm, and ready to send.

## Post 002 — Technical delay

“The API is delayed” is not yet a useful project update.

A strong technical update answers four questions:

- What happened?
- What is the impact?
- What is being done now?
- When is the next update?

Rough:

“deploy failed on migration, rollback worked, no customer impact, checking
before retry”

Ready:

“The deployment failed during the database migration and was rolled back
successfully. There is no customer impact. We’re validating the migration fix
now and will post the next update before retrying.”

The facts did not change. The message became easier to act on.

## Post 003 — Following up

Following up should not require fifteen minutes of rewriting “just checking
in.”

The most effective follow-ups usually contain:

- the decision or item you need,
- the date it affects,
- and the next action you will take.

Instead of:

“Just following up again. Any update?”

Try:

“Could you confirm the direction by tomorrow? That will help me keep the next
step moving without making assumptions.”

Clear is not rude. A useful deadline is not pressure when you explain why it
matters.

## Post 004 — Client setback

Clients rarely panic because a problem exists.

They panic when the update is vague.

A credible setback message needs three things:

1. What changed.
2. What you are doing about it.
3. When they will hear from you again.

Rough:

“launch has to move, final review found issue, tell client without making them
panic”

Ready:

“During final review, we found an issue that needs to be resolved before launch.
We’re addressing it now and will send you the revised launch date by 3 PM
tomorrow.”

Ownership plus a concrete checkpoint builds more confidence than vague
reassurance.

## Post 006 — Ask for a decision

One of the hardest messages at work is asking someone senior to make a decision.

Too soft, and the request disappears.

Too forceful, and it sounds like an escalation.

A balanced version connects the decision to the work it unlocks:

“Could you confirm the direction by tomorrow? That will help me keep the next
step moving without making assumptions.”

The useful pattern is:

decision + reasonable deadline + operational reason.

## Post 007 — The tone test

Before sending a difficult work message, ask three questions:

1. Is the required action obvious?
2. Did I preserve the facts?
3. Does the tone match the relationship?

If any answer is no, another paragraph usually will not fix it.

The message needs a clearer structure.

## Post 008 — Standup update

A useful standup update is not a diary of everything you did.

It should make status, risk, and next action visible.

Rough:

“api thing is delayed, probably tomorrow, not blocked but need design confirm”

Ready:

“The API work is moving, but the final handoff will likely be tomorrow. I’m not
blocked right now—I just need design confirmation before closing it out.”

Same facts. Less ambiguity.

That is the difference between writing more and communicating better.

## Post 009 — Firm does not mean rude

Firm communication is not rude communication.

Rudeness attacks the person.

Firmness makes the constraint and the available choices clear.

Instead of filling a boundary with apologies, try:

“I can take this on, but it will require moving the current deadline or removing
another item from scope. Which option should we use?”

The other person still has agency. The constraint is still real.
